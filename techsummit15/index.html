A Web for Everyone Notes and Resources

A To-Do List for Better Accessibility

  1. Read WebAIM's WCAG 2.0 Checklist. Level A and Level AA items are what we need to aim for.
  2. Use an alt text decision tree to determine appropriate text alternatives for your images (it's all about context).
  3. If you're a mouse user, click on form labels to see if focus moves to the form input.

  7. Refer to the ARIA Design Patterns when you're building a widget.

Current Standards for Higher Ed.

For the last couple of years, the DoED Office for Civil Rights and Department of Justice have been evaluating website accessibility using the Web Content Accessibility Guidelines (WCAG) 2.0, Level AA. Furthermore, Section 508 is being updated to align with the same guidelines. That's what we should use as our benchmark.

Images

Bad or missing alt text is still one of the most common accessibility issues. Always include the alt attribute for images and choose a value that makes sense based on the context of the image. An alt text decision tree can help you figure that out.

Forms

Forms without labels associated with input fields are very difficult (or impossible) for some assistive technology users to navigate. Related form elements (e.g., radio buttons) should also be grouped with fieldset and legend.

Color

Color contrast is a common issue, especially for A&M color palettes. Poor contrast can make it very difficult for low vision users to see your content. It's also important not to use color - by itself - to convey important information (e.g., "required fields are in red" and link text), since not all users will be able to perceive color and/or color differences.

Semantic Structure

Tags like heading, list and table header all convey important information about the structure of content. Assistive technologies will use these tags to help users navigate and understand content. It's important to use the correct tags to mark up content and not just styles to make content look a particular way. Teach content contributors how to add these tags in your CMS.

Keyboard Access

Everything that can be done with a mouse and keyboard should be accessible with a keyboard, alone (Tab, Shift + Tab, arrows, Enter/Space). That means using focus indicators (e.g., a:focus) so it's easy to tell what elements have keyboard focus. And focus order should be logical, which often means stripping out any tabindex values greater than 0.

ARIA

ARIA defines HTML attributes to convey correct role, state, properties, and values. It fills in some of the gaps assistive technology users might experience with native HTML, especially with custom widgets. However, ARIA only changes how assistive technology (more specifically, screen readers) interpret content. You'll still need to handle keyboard access, visibility, properties, etc.

Don't use ARIA when native HTML elements and/or attributes already have your requirements built-in. Refer to ARIA design patterns when you're building custom widgets.
